api-execute-agent.py
Lance les tests pytest avec markers
EXECUTION
Ce que c'est
Lance pytest avec des markers configurables (--markers=smoke), collecte les résultats dans allure-results, et pilote l'exécution de la suite complète ou partielle.
Pourquoi c'est utile
Point d'entrée unique pour déclencher les tests depuis n'importe quel contexte : local, CI/CD, ou autre agent.
Comment vous déclenchez les tests en CI ?
L'api-execute-agent est le point d'entrée unique. Il accepte des markers pytest (smoke, regression, critical) et collecte automatiquement les résultats dans allure-results. Le pipeline CI appelle cet agent avec le marker souhaité. On peut également l'appeler depuis d'autres agents, par exemple smoke-regression-agent l'utilise pour lancer 3 runs consécutifs lors de la détection de flaky tests.

smoke-regression-agent.py
Smoke · Critical · Regression · Go/No-Go
GO/NO-GO
Ce que c'est
Lance des suites ciblées : smoke (5 TCs rapides), critical (9 TCs bloquants), regression (suite complète), et émet un verdict GO / NO-GO pour la mise en production basé sur les résultats.
Pourquoi c'est utile
En pré-prod, on ne lance pas les 50 tests — on lance les 5 smoke tests en 2 minutes. Si ça passe, on déploie. Si un TC critique échoue, NO-GO automatique sans attendre le rapport complet.
Comment vous savez qu'une release peut partir en prod rapidement ?
Le smoke-regression-agent lance d'abord les 5 tests smoke critiques en 2 minutes. S'ils passent tous : GO provisoire. Ensuite les 9 tests critical en 5 minutes. S'ils passent : GO confirmé. Le verdict est automatique avec le détail des TCs en échec. C'est beaucoup plus rapide que d'attendre la suite de régression complète de 30 minutes avant de décider.

🧠 Analyse & Intelligence Artificielle — 13 agents

triage-agent.py
Confidence Scoring · Classification des échecs
CONFIDENCE SCORING
Ce que c'est
Classe chaque échec Allure en : real_bug, flaky, env_issue, false_positive — avec un score de confiance 0.0-1.0. Si confidence < 0.70 → marqué pour revue humaine.
Technique
chat_confident() — retourne {response, confidence, reasoning, needs_human_review}. Pas de classification binaire — chaque décision est probabiliste.
Comment l'agent distingue un vrai bug d'un test flaky ?
Il analyse le message d'erreur, le stack trace et le nom du test. Un AssertionError sur une valeur métier = probable real_bug. Un ConnectionTimeout ou StaleElement = probable flaky. Le Confidence Scoring ajoute "j'en suis sûr à 82%" — si le score est bas (<0.70), l'agent ne décide pas seul et marque pour revue humaine. On automatise les décisions à haute confiance, on escalade les cas douteux.

rca-agent.py
Chain of Thought · Root Cause Analysis
CHAIN OF THOUGHT
Ce que c'est
Remonte jusqu'à la cause racine des échecs. Pas juste "TC-023 a échoué" mais "5 tests échouent parce que le service auth retourne un token expiré depuis la v2.1.0". Groupe les échecs par cause.
Technique
CoT Phase 1 (ÉTAPE 1→2→CONCLUSION) + Structured Output Phase 2 pour le rapport structuré.
C'est quoi Chain of Thought et pourquoi ici ?
Chain of Thought force l'IA à raisonner étape par étape avant de conclure. Sans ça, le LLM saute directement à une conclusion qui peut être fausse. Avec CoT : "ÉTAPE 1 — j'analyse les patterns d'erreur. ÉTAPE 2 — je cherche les corrélations avec les changements récents. CONCLUSION — la cause est X." Ça réduit les erreurs d'analyse de 30-40% sur les diagnostics complexes.

bug-analyzer.py
CoT + Structured Output · Patch automatique
AUTO-PATCH
Ce que c'est
Analyse un échec de test et génère un patch de correction. Avec --apply : applique automatiquement le patch avec backup dans docs/bug-analyzer-backups/.
Technique
CoT Phase 1 + PATCH_SCHEMA : file_path, original_code, fixed_code, safe_to_autofix. Le champ safe_to_autofix empêche l'application des corrections risquées.
Vous n'avez pas peur que l'IA casse du code en appliquant des patches ?
C'est pour ça qu'on a safe_to_autofix. L'agent évalue lui-même si la correction est sans risque. Si la logique est complexe, safe_to_autofix = false et le patch est présenté pour revue sans être appliqué. Avant toute application, le fichier original est sauvegardé. La rollback est instantanée. On automatise uniquement les corrections à faible risque.

tc-generator-agent.py

STRUCTURED OUTPUT
Ce que c'est
Génère des cas de tests complets (Given/When/Then) depuis une User Story. Schema strict : tc_id, us_id, type, priority, title, given, when, then, expected_status, tags.
Pourquoi Structured Output plutôt que du texte libre ?
Sans Structured Output, le LLM répond en texte libre. Extraire tc_id ou expected_status nécessite du parsing regex qui casse dès que l'IA change sa formulation. Avec Structured Output, on impose un schéma JSON — le code reçoit directement un dictionnaire Python propre. C'est la différence entre une API documentée et scraper un site web.

verifier-agent.py
Adversarial Prompting · Vérification par contradiction
ADVERSARIAL
Ce que c'est
Tente de réfuter les résultats produits par les autres agents. Vérifie 5 modes : gherkin, bug, kpi, sync, tc/all. Retourne VALID / INVALID / WARNING.
Comment tu garantis que ton IA ne fabrique pas de fausses alertes ?
Notre verifier-agent utilise l'Adversarial Prompting. Pour chaque résultat critique, un second agent a pour seul rôle de prouver que le premier a tort. Il compare la conclusion avec les données brutes. Si il ne trouve pas d'erreur : VALID. Si il y a une incohérence : INVALID. C'est la logique d'un audit indépendant — on ne laisse jamais un seul agent avoir le dernier mot sur une décision critique.

flaky-agent.py
Détection flaky · Score instabilité · Quarantaine
FLAKY DETECTION
Ce que c'est
Lance N runs consécutifs, calcule le score d'instabilité de chaque TC (flaky si échec sur ≥ 1/3 des runs), émet un Go/No-Go, et peut mettre un test en quarantaine dans Jira automatiquement.
Technique
Seuil FLAKY_THRESHOLD = 0.34. Si TC échoue sur ≥ 34% des runs → flaky. Intégration Jira pour la quarantaine automatique.
Comment vous gérez les tests flaky qui polluent vos résultats de CI ?
Le flaky-agent lance automatiquement 3 runs consécutifs en cas de doute. Si un TC échoue sur au moins 1 run sur 3, il est classé flaky avec un score d'instabilité. L'agent peut automatiquement ouvrir un ticket Jira de quarantaine pour isoler ce test du pipeline principal. Les tests en quarantaine continuent à tourner mais leurs résultats n'impactent pas le verdict du Quality Gate. On ne supprime pas les flaky tests — on les isole le temps de les corriger.

release-advisor-agent.py
Self-Consistency · Décision Go/No-Go
SELF-CONSISTENCY
Ce que c'est
Agrège KPIs + résultats + métriques pour recommander Go/No-Go. Pose la même question 3× avec températures différentes → vote majoritaire.
Pourquoi 3 appels LLM au lieu d'un seul pour la décision de release ?
C'est la technique Self-Consistency. Un seul appel peut varier. En posant la même question 3 fois, on simule 3 experts indépendants. Si 3/3 disent NO-GO, la décision est solide. Si c'est 2/1, on a un accord majoritaire mais avec un signal de risque. Si c'est 1-1-1, on passe en revue humaine. C'est la logique d'un comité de release à 3 personnes.

PREDICTIVE AI
Ce que c'est

Ce n'est pas de la voyance, c'est de la régression statistique augmentée par du LLM pour les cas ambigus.

OBSERVABILITY
Ce que c'est
Surveille les performances des agents IA : durée des appels LLM, taux d'erreur, coût estimé. Lit logs/traces.jsonl écrit automatiquement par tracer.py à chaque appel LLM.

resilience-agent.py
Fallback · Circuit Breaker · 3 états
CIRCUIT BREAKER
Ce que c'est
Si Groq est en panne : basculement automatique Groq → Ollama → Cache → Default. 3 états : CLOSED / OPEN (fail fast après 3 échecs) / HALF_OPEN (1 test après 30s).
C'est quoi la différence entre Circuit Breaker et un simple retry ?
Le retry dit : "essaie 3 fois avec 2s d'attente". Si le service est en panne 5 minutes, tu bloques 6s par appel multiplié par tous tes agents. Le Circuit Breaker dit : "après 3 échecs, je sais que le service est en panne, je ne gaspille plus de temps — je passe au fallback". C'est le pattern standard des microservices resilients, popularisé par Netflix Hystrix.

memory-agent.py
Episodic Memory · Mémoire persistante
EPISODIC MEMORY
Ce que c'est
Chaque run d'agent est sauvegardé comme un "épisode" dans memory/episodes.jsonl. Les agents suivants lisent l'historique : TC-023 était flaky 3 fois → classifié flaky avec plus de confiance.
Comment vos agents s'améliorent avec le temps ?
Via la mémoire épisodique. Chaque exécution est sauvegardée. Quand un agent analyse TC-023 au run suivant, il injecte dans son prompt : "Historique TC-023 : flaky×3, real_bug×1, confiance moyenne 0.72, dernier run 2026-06-08". C'est le Context Engineering — enrichir le contexte de l'IA avec des données pertinentes. Plus l'historique est riche, plus les prédictions sont précises.

prompt-versioning-agent.py
Semver · A/B test · Rollback de prompts
PROMPT VERSIONING
Ce que c'est
Versioning semver (1.0.0 → 1.0.1 → 1.1.0) pour les prompts des agents. Chaque modification crée une version. A/B test compare les scores de confiance entre deux versions. Rollback en une commande.
Comment vous gérez la régression quand vous modifiez un prompt ?
Exactement comme du code : versioning semver. On lance un A/B test qui appelle deux versions du prompt sur les mêmes données et compare les scores de confiance. Si la nouvelle version est moins bonne, rollback en une commande. Tout l'historique est dans Git avec les notes de changement. Un changement de prompt est un diff visible et réversible comme n'importe quel changement de code.

MCP + Playwright · Exploration IA de l'app
MCP PROTOCOL
Ce que c'est
Démarre le serveur Playwright MCP en local, connecte l'IA (Ollama — 100% local) et lui donne accès à un vrai navigateur. L'IA explore l'application comme un utilisateur humain et documente ce qu'elle trouve.
Technique
Model Context Protocol (MCP) — standard Anthropic qui permet à un LLM d'utiliser des outils externes (ici un navigateur Playwright) via une interface standardisée.
C'est quoi MCP et pourquoi vous l'utilisez ?
MCP (Model Context Protocol) est le standard créé par Anthropic pour connecter des LLMs à des outils externes. Le mcp-agent connecte l'IA directement à Playwright via MCP, ce qui lui donne un vrai navigateur contrôlable. L'IA peut cliquer, remplir des formulaires, prendre des screenshots, naviguer — exactement comme un testeur humain. Résultat : elle peut explorer l'application, détecter des bugs, et générer des scénarios de tests basés sur ce qu'elle observe réellement dans l'interface.

Questions Générales sur l'Architecture
Pourquoi Groq et pas OpenAI ?
Groq utilise des puces LPU spécialisées 10-20× plus rapides que les GPU. Pour un pipeline QA en CI, la latence compte : 800ms vs 8s change le temps total. De plus, les APIs Groq et OpenAI sont compatibles (même format /openai/v1/) — on peut basculer sur OpenAI en changeant une variable d'environnement.
Pourquoi deux frameworks — Python pytest-bdd ET JavaScript Playwright ?
Ils couvrent deux périmètres différents. Le framework Python teste les APIs REST (backend) : appels HTTP, codes de retour, payload JSON. Le framework JavaScript teste l'UI (frontend) : navigation, formulaires, assertions visuelles. Les deux partagent la même philosophie BDD Gherkin et la même intégration Jira, mais les technologies sont adaptées à leur couche respective.
Comment vous gérez les données sensibles dans les prompts ?
Les prompts n'envoient jamais de données de production — uniquement des métadonnées : nom du TC, code d'erreur, nom du fichier. Les données sensibles restent sur site. La clé API Groq est dans .env jamais committé dans Git. Pour les contraintes de sécurité strictes, le fallback Ollama permet de tout faire en local sans appel externe.
Quel est le ROI de ce framework ?
Sur un projet de 500 TCs avec 2 runs quotidiens : le triage manuel de 50 échecs prend 1-2h. L'agent fait ça en 2 minutes. Sur un mois : 40-80h économisées rien que sur le triage. La génération de TCs et le patch auto ajoutent 20-30h. Pour un consultant à 600€/j, c'est 15-20k€ de valeur mensuelle que le framework délivre automatiquement.
Quel est votre niveau avec tout ce que vous avez construit ?
Senior QA Automation Engineer + AI Testing Specialist. Ce framework combine trois domaines normalement séparés : l'automatisation QA (pytest-bdd, Playwright, Allure), le génie logiciel (CI/CD, design patterns), et l'ingénierie IA (LLM, prompting avancé, agents).
